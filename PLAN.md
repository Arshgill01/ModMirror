# PLAN.md — ModMirror Build Plan

## Goal

Build ModMirror, a Devvit moderation app that helps Reddit moderator teams find and reduce enforcement inconsistency.

Core pitch:

> ModMirror finds enforcement drift before your users do.

## Product Positioning

ModMirror is not:

- a queue dashboard,
- a Toolbox clone,
- a strike bot,
- an AI moderator,
- a replacement for human judgment.

ModMirror is:

- a moderation consistency layer,
- a policy agreement tool,
- a guided enforcement workflow,
- an audit trail for exceptions and drift.

## MVP Scope

The MVP has four core pieces:

1. Mirror Scan
2. Policy Agreement Flow
3. Apply Policy Action
4. Consistency Nudge + Override Audit

Everything else is out of scope until these work.

## Wave 0 — Research + Scaffold Proof

10. Can a post/comment menu action trigger a form?
11. Can a form chain to another form?
12. Can a menu action access the target post/comment ID and author?
13. Can the app submit a comment to a removed post/comment thread?
14. If not, must the comment be submitted before removal?
15. Can the app send private messages?
16. Can the app add native Mod Notes?
17. Can the app detect moderator permissions for current user?
18. Does playtest work in the target test subreddit?
19. What are the exact commands for dev, upload, and publish in this generated project?

### Deliverables

## Wave 1 — App Skeleton + Data Contracts

### Objective

Create the stable internal structure that all future waves use.

### Deliverables

- `src/shared/schema.ts`
- `src/shared/constants.ts`
- server route structure
- Redis key helper
- basic dashboard shell
- demo mode flag
- health endpoint
- typed API response shapes

### Acceptance Criteria

- Typecheck/build passes.
- Dashboard loads.
- Health endpoint returns app/subreddit context if available.
- Redis key helper is used everywhere.

## Wave 2 — Mirror Scan + Attribution

### Objective

Implement first-run scan and deterministic rule attribution.

### Deliverables

- Fetch mod log.
- Fetch removal reasons.
- Fetch subreddit rules if available.
- Normalize actions.
- Attribution scoring:
  - exact match
  - fuzzy/simple similarity
  - keyword overlap
  - fallback unmatched
- Confidence scoring.
- Drift summary.
- Demo seed scan.

### Acceptance Criteria

- Real scan works on test subreddit, even if sparse.
- Demo scan shows meaningful drift.
- Dashboard displays:
  - scanned actions
  - attributed actions
  - confidence breakdown
  - unmatched actions
  - drift candidates
- No inferred rule is presented as certain unless confidence is high.

## Wave 3 — Policy Agreement Flow

### Objective

Allow mods to create/edit policy ladders per rule.

### Deliverables

- Policy editor UI.
- Create policy from drift candidate.
- Create policy manually.
- Empty policy fallback.
- Store policies in Redis.
- Small subreddit mode.

### Acceptance Criteria

- A mod can create a policy for a rule.
- A mod can edit a policy.
- A rule with no policy routes to creation flow.
- Small subreddit/no-data state is useful, not empty.

## Wave 4 — Apply Policy Action

### Objective

Allow mods to apply a policy to a post/comment.

### Deliverables

- Post/comment menu item.
- Select rule/policy.
- Show recommendation.
- Confirm action.
- Public comment/private message/log-only delivery mode if verified.
- Store action event in Redis.

### Acceptance Criteria

- Menu action appears only for moderators if possible.
- Target post/comment context is captured.
- Recommended action is shown.
- Confirmed action is logged.
- Delivery behavior is stable and documented.

## Wave 5 — Consistency Nudge + Override Audit

### Objective

Warn when the chosen action differs from policy, and record overrides.

### Deliverables

- Compare selected action to policy.
- Show nudge when deviating.
- Require override reason.
- Optional free-text note.
- Store override event.
- Dashboard aggregate override view.

### Acceptance Criteria

- Deviations require reason.
- Overrides appear in dashboard.
- Aggregate analytics work.
- Per-mod analytics are hidden unless permission-gated or omitted.

## Wave 6 — Polish + Submission

### Deliverables

## Deferred Features

Do not build unless MVP is complete:

- full appeal packet generator,
- complex queue dashboard,
- AI rule classifier,
- automatic bans,
- cross-subreddit benchmarking,
- Discord/Slack integrations,
- LLM summaries,
- external analytics service.

## Risks

### Rule Attribution Risk

Historical mod logs may not contain structured rule IDs.

Mitigation:

- deterministic matching,
- confidence levels,
- manual confirmation,
- demo seed mode,
- track ModMirror-created actions going forward.

### Devvit UX Risk

Menu action flows may be constrained.

Mitigation:

- dashboard for rich analysis,
- forms for quick actions,
- document limitations in RESEARCH.md.

### Small Subreddit Risk

Low volume may make drift analysis weak.

Mitigation:

- policy-first mode,
- minimum thresholds,
- demo mode,
- “prevent future drift” framing.

### Submission Risk

Judges may not deeply install the app.

Mitigation:

- polished screenshots,
- seeded demo,
- clear video narrative,
- strong Devpost copy.
